# Task 13 — HTML evaluation report (`HtmlReportRenderer`) — task-08 Phase C3 (part)

An `evaluation.html` rendering of the evaluation report: the Gradle-test-report look,
but with **Evaluation rate sitting next to Success rate** — the task-08 headline
("100% passed / 85% grounded") as one page. Motivated by review of task-12: Gradle's
own `index.html` has **no extension API** (the HTML is generated by Gradle internals
from the binary results), so amending it would mean post-processing private DOM —
brittle across Gradle versions. Instead the harness ships its own HTML through the
pluggable renderer seam (`EvaluationReportRenderer`, ServiceLoader-discovered) that
task-12 introduced for exactly this kind of growth.

> Status: **planned, not implemented.**

## Shape

One new class in `inquisitor-harness-evaluation-report`
(`io.inquisitor.harness.evaluation.report.HtmlReportRenderer`) plus a third line in
the `META-INF/services/…EvaluationReportRenderer` entry. No changes to the model,
the writer, the session listener, or the judge module — the seam absorbs the whole
feature.

- **Self-contained single file**: inline CSS, no JavaScript, no external assets
  (expandable findings via native `<details>/<summary>`). It must open from
  `build/reports/inquisitor/` with no server and no network.
- **Styling**: visually adjacent to Gradle's test report (its muted palette, summary
  tiles, tabular layout) so it reads as a sibling of `index.html`, but not a clone of
  its DOM — we own the structure.

## Content

1. **Summary tiles** (like Gradle's counters): scenarios / matched (expectation
   gate), steps recorded / evaluated / not evaluated, **evaluation score**, duration.
   Header + actor/judge lines beneath, as in the Markdown headline.
2. **One table per bucket** (explicit / cucumber / intent / …), each row a scenario
   instance:

   | Scenario | Expected | Steps | **Success rate** | **Evaluation rate** | Matched |

   - **Success rate** = PASS-outcome steps / recorded steps — Gradle's notion,
     computed from the actor's verdicts. Expectation semantics stay out of it and
     live in **Matched** (a FAIL-expected scenario with a detected fault shows a low
     success rate *and* Matched ✓ — that contrast is informative, not a bug).
   - **Evaluation rate** = mean judge score over the scenario's evaluated steps
     (`—` when everything was `NOT_EVALUATED`).
   - Bucket header carries the bucket's gate + both aggregate rates.
3. **Findings, expandable**: for every non-`GROUNDED` evaluated step, a
   `<details>` block under its bucket table — actor verdict/reasoning/evidence, the
   judge's findings, and the tool trace in a `<pre>`. Same content as the Markdown
   findings, browsable.
4. **Escaping is mandatory**: reasoning, evidence, feedback and the trace all carry
   JSON/HTML-hostile characters (`<`, `>`, `&`, quotes) straight from live systems —
   every interpolated value goes through one HTML-escape helper. A test pins this
   with a `<script>`-shaped payload.

## Decisions

- **Still `StringBuilder`, not a template engine.** One self-contained page with
  simple repetition doesn't justify a consumer-test-classpath dependency; the
  decision trigger recorded in `docs/decisions.md` stands (JMustache the moment the
  formats sprawl or a designer needs to edit layout without recompiling). Revisit
  here if the HTML exceeds a few hundred lines of renderer code.
- **The plugin's console echo also links `evaluation.html`** when present (one line
  in `EvaluationReportEcho`; the file name is static, like `evaluation.md`).
- Bucket/scenario ordering, keys, and instance-splitting come from
  `EvaluationReport` unchanged — this is a pure view.

## Tests

- `HtmlReportRendererTest`: renders the writer test's sample report — asserts the
  summary tiles (score, gate), one table per bucket, a row with success rate ≠
  evaluation rate (the contradicted-PASS case: 100% success / 50% evaluation), a
  `MISSED DETECTION`-flavored row, the `<details>` finding, and the escaping payload.
- `EvaluationReportWriterTest.discoversBothBuiltInRenderers…` becomes three files
  (`containsExactlyInAnyOrder` gains `evaluation.html`).
- Plugin functional test: the canned report gains an `evaluation.html`; assert the
  echo links it.

## Out of scope (rest of C3)

- README verified-models table generated from `evaluation.json`; `benchmark.yml`
  multi-config runs; publishing the plugin/modules.
- Post-processing Gradle's `index.html` (ruled out — no stable surface).
- Charts/JS interactivity.

## Verification

- `./gradlew build` — new renderer tests green; nothing else changes.
- A real `:inquisitor-demo:evaluate` run against the local models →
  `build/reports/inquisitor/evaluation.html` opens standalone, shows the cucumber
  bucket's success/evaluation-rate contrast and the fault suite's low-success/matched
  rows.
